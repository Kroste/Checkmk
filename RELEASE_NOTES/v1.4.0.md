Version 1.4.0 bringt zwei neue Sichten (Dashboard, Auto-Scroll auf frische
CRITs), eine zentrale Host-→-Domain-Zuordnung für DMZ-Rechner, einen
SSH-Client mit Auto-Wahl RDP/SSH je nach OS und die dazugehörige
Login-Verwaltung. Dazu ein wichtiger Bugfix am Startverhalten.

## Neu

**Dashboard als vierter Tab**

Für jeden angelegten Favoriten eine Kachel mit Hosts-Zahl + Service-
Aggregat (CRIT / WARN / OK / UNK). Kachel-Unterrand in Ampelfarbe
(schlechtester Service-Status im Scope). Klick auf eine Kachel aktiviert
diesen Favoriten und wechselt in den Status-Tab. Nutzt die serverseitige
Livestatus-Query aus 1.3.0 — pro Kachel ein schlanker Call.

**Auto-Scroll zum neu aufgetauchten CRIT**

Wenn seit dem letzten Refresh ein Service NEU CRIT wird, scrollt das
Status-Grid dorthin und markiert die Zeile. Kalt-Start meldet nichts —
nur echte Änderungen.

**Zentrale Host-→-Domain-Zuordnung**

Rechner in `dmz.lhp.intern` sind über den Kurznamen nicht erreichbar.
Ab jetzt gibt es eine zentrale `hosts.json` auf dem Fileshare — alle
Cockpit-Nutzer im Fachbereich teilen die Zuordnung. Ping, RDP und SSH
verwenden automatisch den **FQDN** (`host.domain.tld`).

- Ohne Eintrag: Fallback auf Bootstrap-Default (aktuell `lhp.intern`).
- Datei: `\\Samba01\...\hosts.json`, Pfad in `bootstrap.json`
  überschreibbar (`sharedHostsPath`).
- Kein Verschlüsselungs-Overhead — sind Metadaten, keine Secrets.

**SSH-Client + Auto-Wahl RDP/SSH**

Neuer Kontextmenü-Eintrag **„Remote-Shell öffnen (auto: RDP/SSH)"**
wählt anhand der OS-Familie (aus dem Check_MK-Agent-Service):

- Windows → `mstsc /v:fqdn`
- Linux / OS unbekannt → `cmd /k ssh user@fqdn`

Zusätzlich stehen **„RDP öffnen"** und **„SSH öffnen"** weiterhin
explizit im Menü.

**SSH-Login-Verwaltung**

Pro Host lassen sich Anmeldename und (optional) Passwort speichern.
Ablage user-lokal unter `%APPDATA%\Kroste\Checkmk\ssh-creds.json`.
Passwörter DPAPI-verschlüsselt (CurrentUser-Scope).

Beim SSH-Start wird ein gespeichertes Passwort für **30 Sekunden** in
die Zwischenablage kopiert — der SSH-Prompt kommt, du fügst per
Rechtsklick in cmd ein. Danach wird die Zwischenablage automatisch
geleert. Grund: OpenSSH (Windows-Standard) nimmt keine Passwörter per
CLI-Argument.

**Host-Einstellungen-Dialog**

Rechtsklick auf einen Host → **„Host-Einstellungen (Domain/SSH)…"**
öffnet einen Dialog mit drei Feldern: Domain, SSH-Anmeldename,
SSH-Passwort. Domain schreibt in die zentrale Datei, SSH-Kram lokal.

Leeres Passwort beim Speichern **löscht ein bereits gespeichertes nicht**
(nützlich, wenn nur der Name geändert wird). Nur wenn Name UND Passwort
leer sind, wird der ganze SSH-Eintrag entfernt.

## Behoben

**NullReferenceException beim App-Start**

Wenn `statusview.json` mit `AutoRefresh=true` gespeichert war,
crashte die App beim Ctor: die generierten Setter für
`AutoRefresh`/`RefreshSeconds` liefen vor der Initialisierung des
Timer-Felds. Fix: Timer zuerst anlegen, State danach laden.

## Bekannte Grenzen SSH-Passwort-Transport

- Der Clipboard-Weg funktioniert, aber überschreibt für 30 s deine
  aktuelle Zwischenablage. Bewusster Trade-off, weil das die einzige
  Setup-freie Lösung auf Windows-Standard-OpenSSH ist. Wer stattdessen
  `plink` (PuTTY) mit `-pw` bevorzugt, sagt Bescheid — kann ergänzt
  werden.
- Nach 30 s wird die Zwischenablage blind geleert (Avalonia 12s
  `IClipboard` hat kein `GetTextAsync`). Bei kurzer TTL akzeptables
  Risiko.

## Update-Hinweis

Kein Setup-Aufwand. `settings.json`, `filter.json`, `bootstrap.json` und
`statusview.json` bleiben unverändert nutzbar. Beim ersten Start ergänzt
das Cockpit `bootstrap.json` um die neuen Felder `sharedHostsPath` und
`hostDefaultDomain` mit den Defaults für Samba01 bzw. `lhp.intern`.
